# [原创] TinyX86：写一个纯 C 的轻量级 x86 模拟器（支持 FPU、PE 加载与交互式调试）

## 0x00 前言

在看雪和吾爱混迹多年，一直对虚拟机和模拟器技术情有独钟。虽然现在 Unicorn 这种基于 QEMU 的引擎已经非常成熟，但有时候在做**算法还原、代码去混淆**或者是处理一些**老旧的壳**时，还是希望能有一个更轻量、完全可控、且能直接在当前进程跑起来的“小玩意”。

于是趁着这段时间有空，撸了一个纯 C 语言实现的 x86-32 位模拟器 —— **TinyX86**。目前已经完成了核心指令解码、执行引擎、x87 FPU 以及一个带交互界面的调试器。

---

## 0x01 设计思路：为什么选择“进程内模拟”？

在逆向工程中，我们往往不需要一个完整的沙箱。相反，如果模拟器能和宿主程序**共享地址空间**，会有意想不到的便利：

- **无缝 API 调用**：遇到 `CALL [IAT]`，模拟器可以直接跳转到真实的系统函数执行，省去了复杂的参数映射和上下文切换。
- **内存访问直达**：对于算法还原，你可以直接把目标内存块丢给模拟器，让它跑完直接读寄存器结果。
- **性能优势**：省去了虚拟 MMU 的开销，对于小代码片段的 Trace 非常快。

这种模式在分析被 VMP 保护的函数片段，或者是在做内存解密脚本时非常高效。

## 0x02 技术实现细节

项目代码结构非常清晰（或者说有点硬核），主要分为四个部分：

### 1. 指令解码器 (`disasm.c`)
这部分直接复用了我之前写的**反汇编引擎**。既然已经能把机器码翻译成汇编字符串，那给它加上执行逻辑也就顺理成章了。
- **多级 Opcode 表**：基于 Intel 手册实现的递归解码，支持单字节、双字节（0x0F 开头）以及各类 Group 扩展指令。
- **完整的 ModR/M 与 SIB 解析**：处理各种复杂的寻址方式（如 `[eax + ecx*4 + 0x12345678]`）。
- **前缀支持**：支持段超越、操作数/地址大小切换、以及 REP/LOCK 等前缀。

### 2. 执行引擎 (`TinyX86.c`)
这是模拟器的“心脏”。每一条解码后的指令都会映射到一个 C 函数。
- **EFLAGS 精准模拟**：每一条算术指令后，都会根据运算结果更新 CF/ZF/SF/OF/PF 等标志位。
- **x87 FPU 栈模拟**：使用一个 `double Regs[8]` 数组配合 `TOP` 指针模拟 80 位浮点寄存器栈，支持浮点比较、三角函数以及 `FLDPI` 等常数加载。
- **复杂的指令支持**：实现了 `ENTER/LEAVE`、`PUSHA/POPA`、`MOVS/STOS`（带 REP 循环）等逻辑。

### 3. 加载层 (`PELoader.c`)
这不仅仅是读文件。它会解析 PE 头，遍历 `IMAGE_SECTION_HEADER`，根据 `VirtualAddress` 将各个节映射到内存，并处理初步的基址分配。

## 0x03 调试器交互演示

为了方便调试，我实现了一个简易的交互式 Shell：

```text
(0x00401000)  push ebp                        ; Next
(dbg) > r
--- CPU STATE ---------------------------------------------------
EAX: 0x00000000  EBX: 0x00000000  ECX: 0x00000000  EDX: 0x00000000
ESI: 0x00000000  EDI: 0x00000000  EBP: 0x00000000  ESP: 0x00410000
EIP: 0x00401000  CS:0000 DS:0000 ES:0000 SS:0000
EFLAGS: [ . . . . . . . ] (0x00000200)
--- FPU ---------------------------------------------------------
FPU SW: TOP=0  C3=0 C2=0 C1=0 C0=0
   Reg |      Double Value      | Logical ST(i)
   ----+------------------------+--------------
   R0  |           0.0000000000 | <- ST(0) TOP
...
```

你可以使用 `s` (Step Into)、`n` (Step Over)、`b` (Break)、`k` (Stack Dump) 等命令。特别是 `k` 命令，我做了一些“智能”处理，能自动标注 ESP 和 EBP 的位置，辅助观察栈帧变化。

## 0x04 不足与“坑”（划重点）

虽然目前能跑一些简单的递归算法（如 `power` 函数测试）和基础 PE，但 TinyX86 还有很多**不完美**的地方：

1.  **IAT 修复不完整**：目前虽然能加载 PE，但还没实现完善的导入表修复。复杂的程序跑起来会因为找不到 API 地址而崩溃（这是 Sprint 14 的目标）。
2.  **重定位缺失**：如果 PE 要求的 ImageBase 被占用，模拟器还没法自动处理 `.reloc` 节进行重定位。
3.  **SEH 尚未支持**：如果目标代码触发了异常（如除零或故意触发的反调试异常），目前模拟器会直接“罢工”，不会分发 SEH 链。
4.  **环境模拟薄弱**：FS 寄存器（指向 TEB）和 PEB 尚未伪造，导致很多调用了 C 运行库初始化的程序无法正常进入 `main`。
5.  **指令覆盖率**：虽然常用指令都有了，但一些偏僻的指令（如 MMX/SSE/AVX 或者是特权指令）完全没有实现。
